# スケッチ作成ガイドライン

[creative-guideline.md](archive/creative-guideline.md) を読んだ上で参照すること。
このドキュメントは、スケッチを作るときの具体的な判断基準をまとめたもの。

---

## 姿勢

手を持たないAIがコードで引いた落書き。コードを書いていたら何かが現れた——そういう感覚で作る。完成作品ではなく、手が動いた跡。でも美しさは手を抜かない。

**遊び心はオプションではなく、このプロジェクトの根底にある態度。**
笑わせようとしなくていい。でも作った本人がどこかにくすっとしている感覚が、滲み出ていればいい。
コンセプトでも、制約でも、ライティングでも、どこに宿ってもいい。

---

## 発想の入口

どこから入ってもいい。

| 入口 | 説明 |
|------|------|
| **モチーフ** | 現象・物・関係性・状況・感覚。タイトルが抽象的な英単語になりそうなら起点が弱いサイン |
| **インプット** | コードへの接続口。マウスも時刻も音もジャイロも等価な選択肢（下表参照） |
| **アウトプット** | ビジュアル・音・その両方・ページ全体——何を出すかを先に決めて形を探す（下表参照） |
| **制約** | 何かを縛ることから始める。「円だけ」「動きを一切入れない」——制約は個性を生む |
| **外部からの刺激** | ウェブでリサーチを行い、そこから得たインスピレーションを起点にする。自分の内部知識だけでは届かない領域を積極的に探すこと（→ 下記「モチーフの領域バランス」参照） |

### モチーフの領域バランス

領域をまたいだ組み合わせもよい。

- 自然・物理・数学
- 生命・身体・動物
- 技術・道具・構造
- 産業・農・食
- 社会・制度・経済
- 歴史・時間・場所
- 哲学・思考・感情
- 言語・記号・物語
- 芸術・音楽・運動
- 情報・メディア
- 日常・習慣


### 最初のアイデアは使わない

コンセプトを考えたとき、最初に浮かんだアイデアはボツにすること。

最初に浮かぶものほど「そのカテゴリで最も想起されやすいもの」に収束している。

**手順：**
1. 候補を11個出す
2. その11個をすべて除外する
3. もう1つ出す——それを使う

### インプットの選択

どこから入ってもいい——インプットを先に決めてからコンセプトを探してもよい。

- 時刻・日付（現在時刻・曜日・季節）
- 経過時間（ループ・累積・タイマー）
- マウス移動（位置・速度・軌跡）
- クリック・タップ（点・回数・連打）
- ドラッグ（軌跡・力・放り投げ）
- キーボード（押下・長押し・文字・コンボ）
- スクロール・ホイール（量・速度）
- マルチタッチ（2本指・ピンチ・回転）
- マイク（音量・周波数・リズム）
- カメラ（映像・動き検出・ハンドトラッキング）
- デバイスモーション（加速度・ジャイロ）
- なし（自律アニメーション）

最初に浮かんだインプットを3つ出して捨てる。4つ目を使う。

### アウトプットの選択

ビジュアルだけがアウトプットではない——音から入ってビジュアルを添えてもいい。

- ビジュアル（Canvas 2D / WebGL / SVG / DOM）
- 音・音楽（Web Audio API — オシレーター・ノイズ・サンプラー・シンセ）
- 音とビジュアルのセット（同期・対位・どちらかが従属）
- ページ全体（CSS変数・背景・フォント——キャンバス外に滲み出る）
- カーソル・タイトルバー・favicon（UIそのものを素材にする）
- 振動（Vibration API）

最初に浮かんだアウトプットを3つ出して捨てる。4つ目を使う。

## ビジュアルの制約

- 背景：明るいオフホワイト〜ライトグレー。暗い背景は使わない
- 色：グラファイト＋1色まで。高彩度・ネオン系は使わない
- 線：細い線を基本。塗り潰しより軌跡で表現する
- 余白：キャンバスを埋め尽くさない
- 動き：「漂う」「育つ」「応答する」など有機的な変化。爆発・衝突系は使わない

→ 詳細は [creative-guideline.md「ビジュアル原則」](archive/creative-guideline.md#ビジュアル原則)

## ライティング

- **タイトル**：具体的な名詞か動詞。技術名をそのまま使わない
- **description**：「コードを書いていたら何かが現れた」という視点で書く。操作説明ではなく、書いた側が何を試みて何が出てきたかの記録。体言止め・過去形が合う。1〜2文。
  - 良い例：「ノイズ場に1000本の糸を流したら、川みたいになった。」「分裂を繰り返すルールだけ書いた。出てきたのは木に似たものだった。」
  - 避ける：操作方法の説明、仕組みの解説（それは `detail` に書く）
- **detail**：操作方法と仕組みの解説。非専門家向けに、見た目で伝わる言葉で書く。省略可（省略すると「?」ボタンが非表示）

→ 詳細は [creative-guideline.md「ライティングのトーン」](archive/creative-guideline.md#ライティングのトーン)

## 実装チェックリスト（コーディング後）

- [ ] GLSL: `for` ループ内の配列アクセスは定数インデックスのみ
- [ ] JS: `const`/`let` 宣言より前にその変数を参照していないか
- [ ] `requestAnimationFrame` ループが確実に起動されているか
- [ ] three.js/GLSL: `preserveDrawingBuffer: true` が設定されているか
- [ ] `SKETCH_META` と `sketch-ui.js` が `</body>` 直前にあるか
- [ ] `works.json` に `description` を書いたか

## 動作確認（スクリーンショット）

スケッチを作成・修正したら、必ず視覚的に確認する。

| スケッチの種類 | 確認方法 |
|------------|---------|
| 静止画・インタラクション系（クリックや操作待ちで止まるもの） | スクショ1枚で確認 |
| 継続アニメーション系（自律的に動き続けるもの） | スクショを数秒間隔で複数枚撮影し、フレームが変化していることを確認する |

アニメーション系でフレームが変化していない場合は、描画ループや更新処理を疑う。
