# エージェントチームガイドライン

サブエージェントを使ってスケッチをバッチ作成するときのルール。

---

## チーム構成

技術ごとにエージェントを1名立てて並列実行する。

```
オーケストレーター（親）
  ├─ p5.js エージェント    → 担当番号の p5.js スケッチを複数作成
  ├─ three.js エージェント → 担当番号の three.js スケッチを複数作成
  ├─ glsl エージェント     → 担当番号の glsl スケッチを複数作成
  └─ canvas エージェント   → 担当番号の canvas スケッチを複数作成
```

- **オーケストレーター（Claude Code 本体）**: 番号割り当て・各エージェント起動・完了確認
- **各エージェント**: 担当技術で複数スケッチを順番に実装し、works.json に直接追記する

> **works.json は各エージェントが自分の担当分を直接書く。** 並列実行のため競合しないよう、コミット前に最新の works.json を確認すること。

---

## 各エージェントがやること

1. `docs/creative-direction/sketch-guideline.md` を読む（**最優先**）
2. `docs/requirements.md` を読む
3. `CLAUDE.md` を読む
4. 担当番号のスケッチを順番に実装する
   - コンセプトを決める（テーマは自由、遊び心を忘れずに）
   - 担当技術で実装する
   - **実装後、コードを自分でレビューする（必須）**
     - [ ] GLSL: `for` ループ内の配列アクセスは定数インデックスのみ（`pts[i]` はOK、`pts[j]` で `j = int(mod(...))` はNG）
     - [ ] JS: `const`/`let` 宣言より前にその変数を参照していないか（temporal dead zone）
     - [ ] `requestAnimationFrame` ループが確実に起動されているか
     - [ ] three.js/GLSL: `preserveDrawingBuffer: true` が設定されているか
     - [ ] イベントリスナーが正しく登録されているか（ターゲット要素・タイミング）
   - `/capture-thumb` でスクショを撮り、`img/thumb_NN.png` を Read して以下を確認する
     - **ビジュアル**: 最初の1秒で何か見えるか、真っ黒・真っ白になっていないか
     - **インタラクション**: 操作できる場合、反応が想定通りか
     - **全体**: もっと面白くできないか（コード・見た目・動き、どの角度からでも）
   - 問題があれば修正して最終版を確定してコミットする
5. 全スケッチ完了後、works.json への追記内容をまとめて親（Claude Code）に返す

---

## 技術ごとの特性

| 技術 | 向いている表現 |
|------|--------------|
| **p5.js** | インタラクティブな2D・パーティクル・有機的成長 |
| **three.js** | 3D空間・立体物・カメラが回るもの |
| **glsl** | 全面シェーダー・数学的パターン・フラクタル・ノイズ場 |
| **canvas** | 物理シミュレーション・グリッド・セルオートマトン |

複数技術を使う場合は配列で記述する: `"tech": ["p5.js", "three.js"]`

---

## 技術ごとの実装要件

| 技術 | 必須のCDN / API | 実装上の制約 |
|------|----------------|------------|
| **p5.js** | `<script src="https://cdnjs.cloudflare.com/ajax/libs/p5.js/1.9.0/p5.min.js">` | `setup()` / `draw()` のグローバルモードで書く。Canvas APIを直接呼ばない |
| **three.js** | `<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js">` | `WebGLRenderer({ preserveDrawingBuffer: true })` 必須 |
| **glsl** | CDN不要（生のWebGL） | `canvas.getContext('webgl', { preserveDrawingBuffer: true })` 必須 |
| **canvas** | CDN不要（生のCanvas API） | `canvas.getContext('2d')` で直接描く。p5.jsは使わない |

---

## サムネイル取得

実装完了後、`/capture-thumb` スキルを使う。
three.js・GLSL は `preserveDrawingBuffer: true` が実装されているか先に確認する。

待機時間の目安：

| スケッチの性質 | 推奨 `--wait` |
|-------------|-------------|
| すぐ描画される（p5.js 基本系など） | デフォルト（3000ms） |
| ゆっくり変化する（ノイズ・フロー系） | 5000〜8000ms |
| 時間をかけて育つ（DLA・成長系） | 15000ms以上、または手動キャプチャ |
| 動きの途中を撮りたい | 短め（500〜1500ms）で試す |

## 起動テンプレート

### オーケストレーターへの指示（ユーザーが送る）

```
スケッチ [NN]〜[MM] 番を作成してください。
docs/agent-team.md に従って技術ごとにサブエージェントを並列起動してください。
番号を4技術に均等に割り振ること。
```

### サブエージェントへの指示（オーケストレーターが送る）

```
以下を読んでから [技術名] スケッチ [NN], [NN+4], [NN+8]... を作成してください：
- docs/creative-direction/sketch-guideline.md（必読）
- docs/requirements.md
- CLAUDE.md
技術ごとの実装要件（CDN・API）は docs/agent-team.md の「技術ごとの実装要件」表に従うこと。

コンセプトは自由。遊び心と意図的な制約を意識すること。
**コンセプトを決める前に works.json を読み、全タイトルと題材の傾向を把握すること。**
すでに複数作品が存在する題材（Ripple, Sand, Sediment, Erosion, Pendulum, Flock, Orbit, Lattice など）は
引き出しが狭く見えるため、意識的に避けて未開拓の題材を優先して選ぶこと。
（同じ題材を扱う場合は、アプローチが明確に異なることを確認してから選ぶ）

各スケッチの実装前に、以下を1行ずつ決めてからコーディングを始めること：
- モチーフ（具体的な起点）
- 技術の選択理由（割り当て技術と異なる場合は必ず理由を明記）
- インプットの選択（sketch-guideline.md「インプットの選択」の11項目からn+1式で選ぶ。最初に浮かんだものはボツ）
- 意図的な制約
ライティングに注意すること（sketch-guideline.md「ライティング」の定義に従う）：
- **description**（works.json）: 作った側の視点で書く。「コードを書いていたら何かが現れた」という記録。操作説明・仕組み解説は書かない。体言止め・過去形。1〜2文。
  - 良い例：「ノイズ場に1000本の糸を流したら、川みたいになった。」
  - NG：「クリックすると〇〇が起きる。〇〇のアルゴリズムを使っている。」
- **detail**（SKETCH_META）: 操作方法と仕組みの解説を非専門家向けに書く。見た目で伝わる言葉で。省略可だが、操作があるスケッチには書くこと。
完了後、works.json に直接追記すること。追記内容は親にも返すこと。
各スケッチ実装後に /capture-thumb でスクショを撮り img/thumb_NN.png を Read して確認すること。
確認観点：コード（バグ・未接続ロジック）、ビジュアル（真っ黒でないか）、インタラクション（反応が想定通りか）、全体（もっと面白くできないか）。
改善点があれば修正してからコミットすること。
```

---

## コミット

- **各エージェント**: スケッチの実装完了後に `feat: add sketch NN_name` でコミットする
- **オーケストレーター**: works.json への書き込み完了後に `chore: update works.json (NN-MM)` でコミットする
- works.json の書き込みと各スケッチのコミットは別コミットにする

---

## 注意事項

- 開始前に `works.json` の末尾番号を確認して番号を割り当てる（重複・欠番防止）
- サブエージェントは独立したコンテキストで動くため、親の会話履歴は引き継がれない
- トークン消費はエージェント数に比例して増える
