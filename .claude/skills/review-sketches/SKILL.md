# Review Sketches

`debug_comments.json` を読み込み、ユーザーからの未対応FBを確認して各スケッチのコードを読み、修正方針を立てるスキル。

## ワークフロー

### 1. debug_comments.json を読む

プロジェクトルートの `debug_comments.json` を Read ツールで読む。形式：

```json
{
  "05": [
    { "by": "user", "text": "パーティクル多すぎ、重い" },
    { "by": "ai",   "text": "パーティクル数を上限200に制限（1行）" },
    { "by": "user", "text": "まだ重い" }
  ],
  "09": [
    { "by": "user", "text": "色が単調、HSBで揺らしたい" }
  ]
}
```

- ファイルが存在しない、または `{}` の場合はその旨を伝えて終了
- **未対応** = スレッドの最後のエントリが `"by": "user"` のもの
- **対応済** = 最後のエントリが `"by": "ai"` のもの（スキップ）

### 2. works.json でスケッチのパスを解決する

`works.json` を Read ツールで読み、未対応FBがある番号の `path` フィールドを特定する。

```json
{ "num": "05", "path": "sketches/05_particle_storm/", ... }
```

### 3. 各スケッチのコードを読んで修正方針を出力

未対応FBがある各スケッチについて:
- `works.json` で解決したパスの `index.html` を Read ツールで読む
- スレッド全体（過去のやり取りを含む）とコードを照合して修正方針を考える

### 4. 出力フォーマット

```
## NN — タイトル
> 最新のユーザーコメント（原文）

修正方針:
- 具体的な変更点1（対象ファイル・行数の目安）
- 具体的な変更点2
```

### 5. 修正を実施する

ユーザーから「修正して」と指示があった場合:
- 各スケッチのコードを修正する
- 修正後、`debug_comments.json` の該当スレッドに AI の返答を追記する

```json
{ "by": "ai", "text": "パーティクル上限を1000→200に変更（index.html 1行）" }
```

返答の形式: `何を変えたか（対象ファイル、変更規模の目安）` を1行で。コードの詳細は書かない。

## 注意事項

- デフォルトは修正方針の出力のみ。実装は別途ユーザーの指示を待つ
- 対応済み（最後が `ai`）のスレッドはスキップ
- `debug_comments.json` への追記は Write/Edit ツールで行う（ファイル全体を上書き）
